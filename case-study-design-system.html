01 — Overview

Context & background

The product had grown through acquisition — six separate product lines, each with its own design language, component library, and interaction patterns. Users moving between modules reported confusion, and engineers were maintaining four separate button implementations.

I was brought in as Principal UX Designer to lead the design unification effort: define a shared system, gain buy-in across squads, and ship it without blocking ongoing product work.

02 — Problem

Defining the problem

Before designing anything, I needed to understand the true scale of fragmentation. I ran an audit across all six product lines and catalogued every component — buttons, form fields, data tables, navigation patterns, modals.

"We had 14 different button styles across the platform. Not because they meant different things — but because no one had ever talked to each other."

The audit revealed that 73% of components existed in two or more variants with no semantic distinction. This created real user confusion: the same action looked different depending on which module the user was in.

Before
  • 14 button variants with no semantic logic
  • 4 separate component libraries in Figma
  • No shared vocabulary between teams
  • Inconsistent spacing, type, and colour
  • Handoff process took 5+ days per sprint
After
  • 3 intentional button variants with clear roles
  • Single source of truth in Figma + Storybook
  • Shared token system across design and code
  • Consistent 8pt grid and type scale
  • Handoff reduced to same-day turnaround

04 — Process

How we designed and shipped it

Rather than designing the system in isolation and pushing it to teams, I ran a federated model from day one: each product squad had a design system representative who contributed to decisions and built early ownership.

1

Component audit & taxonomy

Catalogued every UI element across the platform. Named them consistently, mapped overlaps, built a shared vocabulary before touching any pixels.

2

Token system definition

Worked with engineering to define a design token architecture covering colour, spacing, typography, and border radius — single source of truth for Figma and code.

3

Pilot with one squad

Rather than a big-bang rollout, we piloted with the most receptive team first. Their feedback stress-tested the system and created internal advocates.

4

Documentation & office hours

Built a living documentation site and ran weekly office hours. Adoption isn't a launch — it's a service.

5

Governance model

Established a contribution model so teams could propose new components, with clear acceptance criteria and a review process.

07 — Learnings

What I'd do differently

Design systems are products, and like any product they require ongoing investment. The biggest lesson was that the system itself wasn't the hard part — the organisational change was.

↗

Start with governance, not components

We spent the first two months designing components before we had alignment on how decisions would be made. Establishing governance earlier would have saved significant rework.

↗

Make the ROI visible earlier

Stakeholders needed quantitative justification. I should have instrumented the "before" state more rigorously so we could show cleaner before/after data at launch.

↗

Celebrate contributions publicly

Teams contribute more when their work is visible. A regular changelog and contributor spotlight drove unexpected adoption momentum.
